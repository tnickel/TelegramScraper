1 Projektübersicht & Motivation

1.1 Das Problem der Chat-Flut

In modernen Tech-Communities auf Telegram werden täglich Hunderte von Softwarepaketen, Einstellungsdateien und Erfahrungsberichte geteilt. Für Forex-Trading-Communitys ist dies besonders ausgeprägt: Ein einzelner Gruppenkanal kann über 200.000 Nachrichten angesammelt haben, in denen Nutzer Expert Advisors (Handelsroboter), deren Einstellungen und Ergebnisse diskutieren.

Für einen menschlichen Analysten ist es praktisch unmöglich, diese Datenmasse zu bewältigen:

⚠️

Das manuelle Problem

200.000 Nachrichten manuell durchzulesen würde bei einem Lesetempo von 1 Minute pro Nachricht über 3.300 Stunden dauern – das entspricht fast 2 Jahren Vollzeitarbeit. Dazu kommen hunderte manuell heruntergeladene und zugeordnete Dateien.

1.2 Die Lösung: Vollautomatisierter Workflow

Der Telegram Chat & Forex EA Extractor ist eine spezialisierte Python-Anwendung, die diesen Prozess vollständig automatisiert:

  • 📥 Automatisches Herunterladen von Chats über Offline-JSON-Export oder Live-API
  • ✂️ Intelligentes Chunking der Chatlogs in KI-optimierte 500-Nachrichten-Portionen
  • 🤖 KI-Analyse mit OpenRouter/Gemini LLMs zur Extraktion von EA-Details
  • 📊 Interaktives Web-Dashboard zur Visualisierung aller gesammelten EAs
  • ⚙️ Automatisierter Backtest-Workflow mit MetaTrader 5 Integration
  • 📁 Intelligente Dateikatalogisierung aller extrahierten Forex-Dateien
⚡

Effizienzgewinn durch KI-Augmented Engineering

Die gesamte Entwicklung dauerte dank KI-unterstützter Programmierung weniger als 4 Stunden. Das System spart dem Anwender das manuelle Sichten von über 200.000 Chatnachrichten sowie das Herunterladen und Zuordnen von Hunderten geteilter EA-Pakete.

2.2 Die 6-Stufen Pipeline

Der gesamte Analyseprozess ist in einen 6-stufigen Automatisierungs-Workflow unterteilt, der entweder komplett automatisch oder manuell Schritt für Schritt über das Web-Dashboard gesteuert werden kann.

1

📥 Telegram-Extraktion

Download der Chat-Daten via JSON-Export oder Live-API (Telethon). Extraktion aller Forex-Dateien (.ex4, .ex5, .mq4, .mq5, .set) und Aufteilung des Chatverlaufs in 500-Nachrichten-Chunks. Skript: telegram_extractor.py / telegram_api_downloader.py

2

🤖 EA-Details Analyse (LLM)

Automatische Analyse der Chat-Chunks durch OpenRouter/Gemini-LLMs. Extraktion von Timeframes, Währungspaaren, Strategie, Risikoeinstufung und Community-Sentiment pro EA. Skript: extract_ea_details_llm.py

3

📊 Dashboard-Update

Import der LLM-Analyse-Ergebnisse in die SQLite-Datenbank und Generierung des interaktiven HTML-Dashboards. Alle EAs werden mit Score, Risiko-Rating und Details angezeigt. Skript: generate_dashboard.py

4

⚙️ Backtest-Konfiguration (Dry-Run)

Filterung der EAs nach Mindest-Score (Standard: 50) und Ausschluss von Scam-EAs. Generierung einer JSON-Konfigurationsdatei für den Java-Backtester mit allen qualifizierten Robotern. Skript: run_batch_backtests.py --dry-run

5

📈 Backtest-Lauf (MT5)

Vollautomatischer Start von MetaTrader 5 Backtests über den Java-Backtester-Adapter. Alle qualifizierten EAs werden mit ihren empfohlenen Währungspaaren und Timeframes getestet. Skript: run_batch_backtests.py

6

🏆 Ergebnis-Dashboard

Aggregierung aller Backtest-Ergebnisse in eine übersichtliche HTML-Übersicht mit Performance-Metriken, Ranking und statistischer Auswertung. Skript: generate_backtest_dashboard.py

2.3 Threading-Modell

Eine der technisch anspruchsvollsten Herausforderungen ist das Management von Concurrency in der GUI. Das System verwendet eine 4-Thread-Architektur:

Thread-Typ Verantwortlichkeit Technologie Besonderheit
GUI-Thread Zeichnen der Tkinter-Fenster, Reaktion auf Benutzerklicks Tkinter Main Loop Liest alle 100ms die Log-Queue ab (poll_logs())
AsyncLoop-Thread Hält den asyncio Event-Loop für Telethon am Leben asyncio, threading Startet beim Programmstart, nimmt Coroutinen per asyncio.run_coroutine_threadsafe() an
Extraktions-Thread Parst JSON-Dateien im Hintergrund (Methode A) threading.Thread Kann gigantische Dateien (mehrere GB) verarbeiten ohne die GUI zu blockieren
2FA-Handshake Thread-sichere Eingabe von Login-Codes und Passwörtern threading.Event GUI öffnet modalen Dialog, Hintergrund-Thread wartet blockierend
💡

Warum diese Komplexität?

Tkinter ist nicht thread-safe. Alle GUI-Updates müssen im Hauptthread stattfinden. Gleichzeitig benötigt Telethon (die Telegram-Bibliothek) einen eigenen asyncio Event-Loop. Die queue.Queue dient als thread-sichere Brücke zwischen beiden Welten.

Aggregator-Service für hunderte KI-Modelle. Standard-Modell: meta-llama/llama-3.1-8b-instruct – kosteneffizient ($0,02 / 1M Tokens), stabil, keine aggressiven Rate-Limits. Alternativ kompatibel mit GPT-4o, Claude 3.5, Gemini Pro.

Das LLM-Ausgabeformat wird durch ein Pydantic-Datenmodell streng typisiert und validiert. Robust gegen malformatierte JSON-Antworten durch dreistufigen Fallback-Parser (direkt → Markdown-Block → Substring-Extraktion).

4.4 extract_ea_details_llm.py – Die KI-Analyse-Engine

Das Kernstück der automatisierten Analyse. Dieses Modul liest die Chat-Chunks, identifiziert relevante EAs und analysiert sie mit einem LLM.

①

EA-Identifikation

Analysiert Dateinamen aus forex_files/ und bereinigt Namen (entfernt Versionsnummern, generische Begriffe). Sucht nur EAs mit mind. 5 Erwähnungen im gesamten Chatverlauf.

②

Keyword-Multi-Mapping

Jede Nachricht wird gegen alle EA-Keywords geprüft. Nachrichten, die mehrere EAs erwähnen, werden allen betroffenen EAs zugeordnet. Optimierung: einmalige Lowercase-Konvertierung aller Nachrichten statt 89M Aufrufe.

③

Batch LLM-Synthese

Für jeden EA: Sammlung der letzten 35 Postings, Zusammenführung zu einem einzigen Kontext-Block, Senden an OpenRouter API. Temperature: 0.2 (deterministisch, strukturiert). Rate-Limiting: 4,5 Sekunden Pause zwischen Anfragen (15 RPM max).

④

Inkrementeller Cache

Nach jeder erfolgreichen LLM-Antwort wird das Ergebnis sofort in ea_insights.json gespeichert. Bei Neustart wird der Cache geladen und die Verarbeitung genau am nächsten unanalysierten EA fortgesetzt.

5 Verwendete Prompts

Das System verwendet mehrere Prompt-Strategien, um KI-Modelle optimal zu nutzen. Alle Prompts sind auf Struktur und Determinismus ausgelegt.

5.1 Chat-Chunk System-Prompt

Jeder Chat-Chunk beginnt mit einem eingebetteten System-Prompt, der dem Analysten (Mensch oder KI) sofort erklärt, was zu tun ist. Dadurch kann der Chunk direkt in ein KI-Tool eingefügt werden.

📝 System-Prompt (Eingebettet in chat_chunk_XXX.txt)

Beachte besonders Diskussionen über Forex-Roboter (EAs), Strategien, .set-Einstellungsdateien, Backtest-Ergebnisse oder Fragen der Nutzer. Jede Nachricht ist wie folgt formatiert: [ID] | [Datum] | [Absender]: Text ======================================================================== [1452] | [2026-06-04T16:45:00] | Max Mustermann: Hier ist die neue Version des Gold-EAs! [Angehängte Forex-Datei: Gold_Pure_V1.ex5] ---------------------------------------- [1453] | [2026-06-04T16:46:00] | Sandra K.: Habe es getestet! XAUUSD M15 läuft gut. 5.2 LLM Analyse-Prompt (OpenRouter)

Der eigentliche LLM-Prompt für die automatische EA-Analyse ist präzise strukturiert, um deterministisch ein valides JSON zurückzubekommen.

🤖 LLM Analyse-Prompt (extract_ea_details_llm.py)
Du bist ein erfahrener Handelsroboter-Experte (Forex EAs). Analysiere die folgenden echten Telegram-Chatlogs über den EA '{ea_name}'. Die Community diskutiert darin über Setups, Einstellungen und Ergebnisse. Antworte AUSSCHLIESSLICH mit einem einzigen, validen JSON-Objekt im folgenden Format: { "description": "Kurzbeschreibung des EA (2-3 Sätze in Deutsch, basierend auf Community-Meinungen)", "timeframes": ["M1", "M15"], "pairs": ["EURUSD", "GBPUSD"], "strategy": "Strategie-Bezeichnung (z.B. Grid / Martingale)", "risk": "Risiko-Einschätzung (z.B. Low, Medium, High)", "warnings": "Kritikpunkte, Risiken oder Warnungen" } Achtung bei Währungspaaren und Timeframes: - Ziehe nur Währungspaare und Zeiteinheiten heraus, die explizit empfohlen werden. - Fasse das Feedback kurz in einer verständlichen deutschen Beschreibung zusammen. - Falls keine Zeiteinheiten oder Paare genannt werden, trage "Keine Nennung" ein. [CHATLOGS START] {postings_text} ← bis zu 35 gefilterte und gereinigte Postings [CHATLOGS ENDE]
🎯

Prompt-Design-Prinzipien

Temperature 0.2: Niedrig für maximale Konsistenz und strukturierte Ausgabe. | Rollendefinition: "Handelsroboter-Experte" führt zu besseren Fachbegriffen. | Negativ-Beispiel: "AUSSCHLIESSLICH JSON" verhindert Prosa-Antworten. | Fallback-Parser: Dreistufig (direkt → Markdown-Block → Substring).

10 KI-Workflow: Chunking & LLM-Analyse

10.1 Das "Lost in the Middle"-Problem

Moderne KI-Sprachmodelle haben zwar wachsende Kontextfenster (teils über 1 Million Tokens), neigen aber dazu, Informationen in der Mitte langer Dokumente zu übersehen. Dieses Phänomen wird als "Lost in the Middle" bezeichnet.

📊 Warum 500 Nachrichten pro Chunk?
graph LR A["❌ 1 Riesen-Chunk\n200.000 Nachrichten\n→ Wichtige Details\n werden übersehen"] B["✅ 400 kleine Chunks\n500 Nachrichten each\n→ Vollständige Analyse\n pro Chunk möglich"] C["⚖️ Optimum gefunden!\n500 Msg = ~15-20k Tokens\nKlein genug für Fokus\nGroß genug für Kontext"] A -->|"Lost in the Middle-Problem"| C B -->|"Bewährte Praxis"| C

10.2 Der Chunking-Prozess im Detail

ParameterWertBegründung
Chunk-Größe 500 Nachrichten Optimaler Kompromiss: Vollständig analysierbar, Kontext über mehrere Stunden erhalten
Reihenfolge Chronologisch (Alt → Neu) Diskussions-Kontext bleibt erhalten (Antworten folgen auf Fragen)
Vorspann System-Prompt in jeder Datei Chunk kann direkt in KI-Tool eingefügt werden, ohne Erklärung
Format [ID] | [Datum] | [Absender]: Text Maschinenlesbar UND menschenlesbar, Zeitstempel für Kontext
Dateinamen chat_chunk_001.txt ... chat_chunk_400.txt Numerisch sortierbar, einfach iterierbar

10.3 LLM-Analyse Performance-Optimierungen

⚡
Einmalige Lowercase-Konvertierung
Alle Nachrichten werden einmalig in Kleinbuchstaben konvertiert und als Tupel gespeichert (original, lowercase). Das verhindert bei 463 EAs × 192.000 Nachrichten 89 Millionen redundante .lower()-Aufrufe.
💾
Inkrementeller Cache
Nach jeder LLM-Antwort wird sofort in ea_insights.json gespeichert. Bei Stromausfall oder Abbruch: Neustart setzt genau am nächsten unanalysierten EA fort (z.B. bei Index 1001, wenn 1000 bereits verarbeitet).
🎯
Batch-Synthese (35 Postings)
Nur die letzten 35 Postings pro EA werden an das LLM gesendet. Dies fokussiert auf aktuelles Community-Feedback, minimiert API-Kosten und vermeidet veraltete Informationen aus frühen Experimentphasen.
🔄
Rate-Limiting
Automatische Pause von 4,5 Sekunden zwischen API-Aufrufen (OpenRouter Free Tier: max. 15 RPM). Verhindert HTTP-429-Fehler und stellt zuverlässige Verarbeitung sicher.

14 Entwickler-Richtlinien

14.1 Kritische Coding-Regeln

🚫

Regel #1: Niemals os.getcwd() für Projektpfade verwenden!

os.getcwd() gibt das Verzeichnis zurück, von dem aus das Skript aufgerufen wird – nicht das Verzeichnis, in dem das Skript liegt. Bei Aufruf aus einem anderen Ordner schlagen alle Dateioperationen fehl.

Python – Korrekte Pfadauflösung (OBLIGATORISCH) 
from pathlib import Path

# ✅ RICHTIG: Skriptrelative Pfade
PROJECT_ROOT = Path(__file__).parent.parent.resolve()
LOG_DIR      = PROJECT_ROOT / "log"
DATA_DIR     = PROJECT_ROOT / "extracted_data"
CONFIG_FILE  = PROJECT_ROOT / "config.local.json"

# ❌ FALSCH: Wird vom Start-Verzeichnis abhängig
# PROJECT_ROOT = Path.cwd()  ← NIEMALS SO!
# PROJECT_ROOT = Path(os.getcwd())  ← NIEMALS SO!
🔒

Regel #2: Lock-Release vor allen I/O-Operationen im HTTP-Server

Netzwerk-Calls und HTTP-Response-Schreibvorgänge dürfen NICHT innerhalb einer pipeline_lock-Sperre stattfinden. Immer: Daten kopieren → Lock freigeben → I/O.

Python – Korrektes Lock-Pattern im HTTP-Server 
# ✅ RICHTIG: Lock vor I/O freigeben
with pipeline_lock:
    data_copy = pipeline_state.copy()  # Daten kopieren
# Lock wurde durch 'with'-Block beendet
response = json.dumps(data_copy)  # I/O AUSSERHALB des Locks
self.wfile.write(response.encode())

# ❌ FALSCH: Lock während I/O gehalten → Deadlock!
# with pipeline_lock:
#     self.wfile.write(json.dumps(pipeline_state).encode())  ← DEADLOCK!
🧵

Regel #3: Tkinter-Widgets nur im Hauptthread modifizieren

Alle GUI-Änderungen müssen über self.log_queue.put() aus Hintergrundthreads kommuniziert werden. poll_logs() im Hauptthread verarbeitet dann die Queue thread-sicher.
